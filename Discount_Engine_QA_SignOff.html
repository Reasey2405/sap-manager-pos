1. Purpose of This Document

This document is a formal QA sign-off package for the Discount Engine developed in the POS Manager system. It explains what was built, how it works, what must be tested, and what criteria must be met before development moves to Phase 2.

Written for QA testers who may not have a software development background. Technical concepts are explained in plain language throughout.

Phase 1 — Current
The Discount Engine is built and tested in the POS Manager only. The POS Manager is always online. No offline terminals are involved yet.
Phase 2 — Future
Once QA confirms Phase 1 is working correctly, the same discount logic will be replicated into the offline POS Terminal.

2. What Is the Discount Engine?

The Discount Engine is the part of the system that decides whether a customer gets a discount, how much that discount is, and which discounts apply when a customer is at the checkout.

When a cashier scans items at the POS terminal, the system sends the receipt to the Discount Engine. The engine runs the receipt through a fixed sequence of evaluation stages — from the broadest filter down to the finest detail — and returns the final discounted total.

2.1 Full Evaluation Flow — Biggest to Smallest

Every receipt is processed through exactly these stages in this order. Stages 1–2 are receipt-level gates. Stages 3–5 loop through each qualifying scheme. Stage 6 resolves competing discounts. Stages 7–9 finalise and package the result.

Stage Level Stage Name What Happens — Plain English
1 Receipt Receipt Validation The receipt is checked before anything else. If it has no lines, or any line has a missing or negative quantity/price, an error is returned immediately. Nothing else runs.
2 Receipt Active Scheme Lookup The engine fetches all ACTIVE schemes within their valid date range. Only these are candidates.
3 Scheme Scope Filter Each candidate scheme is checked against the receipt's POS terminal and group. GLOBAL schemes pass always. POS_TERMINAL and POS_GROUP schemes only pass if the terminal/group matches. Failing schemes are dropped.
4 Scheme Entitlement Filter Each remaining scheme is checked against who the customer is. PUBLIC schemes pass for anyone. CUSTOMER_GROUP, SPECIFIC_CUSTOMER, and COUPON schemes only pass if the receipt matches. Failing schemes are dropped.
5a Rule Rule Sequence Sort Inside each scheme, rules are sorted by their sequence number (lowest first) before evaluation begins.
5b Rule Rule Threshold Gates Each rule's minimum order amount and minimum quantity are checked. If the receipt total is below minOrderAmount, or item count is below minQuantity, the entire rule is skipped.
5c Rule Condition Evaluation ALL conditions on the rule are evaluated: product, category, time of day, day of week, receipt value, quantity, payment method, first purchase, coupon code. Every condition must pass. If any fails, the whole rule is skipped.
5d Receipt Line Line Qualifier Gates For LINE_ITEM and CHEAPEST_ITEM types, each receipt line is checked against minLineTotal and minLineUnitPrice. Lines that don't meet these are silently excluded.
5e Receipt Line Discount Calculation The discount amount is calculated for each qualifying line or the transaction. Rounding is applied. The per-result maxDiscountAmount cap is applied. Zero-discount results are removed.
5f Scheme Per-Scheme Transaction Recalculation (1st) Transaction-level discounts are recalculated so their base reflects the line discounts already calculated in this scheme.
5g Scheme Scheme Total Saving Tally The total saving for this scheme is summed. Used later for combination resolution (best-of, exclusive winner selection).
6 Receipt Combination Resolution Schemes sorted by priority (highest first) and grouped. EXCLUSIVE → highest saving wins, all processing stops. ADDITIVE → all stack. BEST_OF → only highest saving kept. Winning BEST_OF merged with ADDITIVE results.
7 Receipt Final Transaction Recalculation (2nd) recalculateTransactionDiscounts() called again on the fully merged result set. Ensures transaction discounts reflect combined line discounts from ALL schemes.
8 Receipt Receipt Total Cap Ensures total discounts never exceed the original receipt total. If combined discounts would go negative, they are proportionally trimmed.
9 Receipt Response Build Final result assembled: original total, total discount, final payable total, and a line-by-line breakdown of every discount applied.
Key Rule: Stages Never Run Out of Order

The engine always runs these stages in sequence. A receipt that fails Stage 1 never reaches Stage 2. A scheme that fails scope (Stage 3) is never checked for entitlement (Stage 4). A rule that fails threshold gates (Stage 5b) never has its conditions evaluated (Stage 5c).

QA must test failures at EACH stage independently — not just verify the final result is zero.

2.2 Level Summary

Level Applies To What It Means
Receipt The whole transaction Evaluated once per receipt. Affects whether any discount runs at all, or what the final combined result looks like.
Scheme One discount scheme Evaluated once per qualifying scheme. Determines whether the whole scheme applies to this receipt.
Rule One rule within a scheme Evaluated once per rule. A scheme can have multiple rules; each is evaluated independently.
Receipt Line One item line on the receipt Evaluated once per receipt line. Determines whether each individual product qualifies for the discount.
2.3 How a Receipt Is Read by the Discount Engine (Developer Only)

Before evaluation begins, the receipt fields are mapped into the format the engine expects. QA must know which fields drive which discount behaviour.

What This Means for QA

If a receipt field is missing or null, any discount that depends on it will not fire. Voided and Cancelled lines are removed before evaluation — they never receive a discount.

2.3.1 Receipt Header Fields

Receipt Field Used For How the Engine Uses It
posTerminalID Terminal scope check Identifies which POS terminal sent this receipt. Discounts set up for a specific terminal only apply if this matches.
posTerminalID → POS Group lookup Group scope check The system looks up which POS group this terminal belongs to. Discounts set up for a POS group apply if the terminal is in that group. Null if terminal has no group.
cardCode Customer identity check Used to match discounts restricted to a specific named customer.
cardCode → Customer group lookup Customer group check The system looks up which group this customer belongs to. Used for CUSTOMER_GROUP entitlement and condition matching. Null if customer not found.
(always false — hardcoded) First purchase flag Always set to false. FIRST_PURCHASE discounts will never fire through this path.
payments[0].paymentType Payment method check Only the first payment line's type is used. If the receipt has no payments or the first payment has no type, payment-method discounts will not apply.
couponCodes (separate parameter) Coupon check Passed in separately — not stored on the ReceiptDto. If not provided, all coupon-based discounts will not apply.

2.3.2 Receipt Line Fields

Receipt Line Field Used For How the Engine Uses It
id (line ID) Receipt line number Identifies which line the discount is applied to. If null, a counter (1, 2, 3…) is used instead.
itemCode Product matching Used to check if a specific product is required by the discount rule (PRODUCT condition, BUY_X_GET_Y scope).
itemCode → item category lookup Category matching The system looks up the item's group code. Used for CATEGORY condition matching. Null if item not found.
quantity Item count & line total Used to count total items (QUANTITY_THRESHOLD), calculate the line total, and BUY_X_GET_Y trigger counts. Defaults to zero if blank.
unitPrice (or price fallback) Price per item Engine uses unitPrice first; falls back to price field if null. Defaults to zero if both blank.
(auto-calculated) Line total quantity × unitPrice. Calculated by the engine. Used in BASKET_VALUE condition, minLineTotal gate, and all line discount calculations.
documentStatus = Void/Cancel Line exclusion Voided and Cancelled lines are completely removed before the engine runs. They never receive a discount.

2.3.3 Receipt Fields That Do NOT Affect Discounts

Receipt Field Why It Does Not Affect Discounts
receiptNumber Only used after discounts are calculated, when recording which discounts were applied.
discountPercent / totalDiscount (header) Existing POS receipt discount fields. The engine ignores them and calculates its own discounts from scratch.
totalBeforeDiscount / totalAfterDiscount / totalAfterTax Pre-calculated receipt totals. The engine recalculates from receipt lines — it does not trust these.
taxAmount / vatGroupRate / vatGroup Tax is not considered during discount evaluation. The engine works on pre-tax prices.
cardName Customer name. Only the customer code (cardCode) matters for entitlement.
series / SAP document references SAP integration fields. Irrelevant to discount calculation.
payments[1], [2], … (non-first payments) Only the first payment line's type is checked. Additional payment methods are ignored.
Receipt line: totalPrice / discountPercent / uomEntry / uomName Pre-calculated POS values. The engine recalculates from quantity × unitPrice.

Select the Discount Type first — the form shows or hides additional fields depending on the type.

UI SetupEnter the % rate in Discount Value (e.g. 10 = 10%). Set Apply To: TRANSACTION for whole receipt, LINE_ITEM per line, or CHEAPEST_ITEM for the lowest-priced item.

4. How Multiple Discounts Are Combined

When more than one discount qualifies for the same receipt, the Combination Mode set on the General tab determines what happens.

ADDITIVE
All qualifying discounts are applied together and stacked. The customer receives the benefit of every matching discount.
Example: Scheme A gives $5 off, Scheme B gives 10% off. Both apply — customer saves $5 plus 10%.
EXCLUSIVE
Only the single highest-saving EXCLUSIVE scheme applies. When an EXCLUSIVE wins, all other schemes — including ADDITIVE ones at lower priority — are ignored entirely.
Example: Three exclusive discounts qualify ($5, $10, $8). Only the $10 saving applies. All others stop.
BEST_OF
Among all BEST_OF schemes, only the highest saving is applied. The winning BEST_OF is then merged with any ADDITIVE results.
Example: Two Best-Of qualify — one saves $8, the other $12. Only $12 applies. ADDITIVE schemes still stack alongside it.

5. Scheme Priority System

Every discount scheme has a Priority number. The engine uses this number to decide the order in which schemes are evaluated and how they interact when multiple schemes qualify for the same receipt.

5.1 How Priority Works

Higher priority number = evaluated first. The engine sorts all qualifying schemes from highest to lowest priority before resolving combination modes.

Rule Behaviour
Higher number first A scheme with priority 10 is processed before a scheme with priority 5.
Same priority grouped Schemes sharing the same priority number are processed as a group together before moving to the next level.
Null priority = 0 If a scheme has no priority set, it defaults to 0 and is processed last.
EXCLUSIVE halts all If an EXCLUSIVE scheme wins within any priority group, processing stops immediately. No lower-priority schemes are evaluated — even ADDITIVE ones.
BEST_OF is cross-group The best-of winner is tracked across all priority groups and merged with ADDITIVE results at the end.

5.2 Priority Evaluation Flow (Step by Step)

This is exactly what the engine does internally when resolving priority and combination modes:

# What the Engine Does
1 Sort all qualifying schemes by priority (highest number first).
2 Group schemes that share the same priority number into a batch.
3 For the current priority group: check if any scheme is EXCLUSIVE.
4 If EXCLUSIVE found → select the EXCLUSIVE scheme with the highest saving. Return immediately. Stop all further processing.
5 If no EXCLUSIVE → process each scheme by its combination mode: ADDITIVE schemes added to final result. BEST_OF schemes compete — only the highest saving BEST_OF is remembered.
6 Move to the next priority group and repeat steps 3–5.
7 After all groups are processed: merge the winning BEST_OF (if any) into the ADDITIVE results.
8 Run recalculateTransactionDiscounts() on the final merged result set.

6. Line Discount First, Then Transaction Discount

This is one of the most important behaviours to understand and verify. When a receipt has both receipt-line discounts and whole-receipt discounts, the engine applies them in a specific order to ensure the transaction discount is always fair and mathematically correct.

6.1 The Rule

Core Behaviour — Confirmed in Code

Step 1: Receipt-line discounts are calculated first against the original line totals.

Step 2: The engine sums all line discount amounts to get the total line saving.

Step 3: Remaining receipt total = Original receipt total MINUS total receipt-line discounts.

Step 4: The whole-receipt discount is recalculated using this reduced amount — NOT the original receipt total.

Result: The customer never receives a whole-receipt discount on top of an amount that has already been fully discounted at the receipt-line level.

6.2 Worked Example

This example shows exactly how the engine calculates when both receipt-line and whole-receipt discounts are active on the same receipt:

Step Value Notes
Original receipt total $100.00 3 lines: $40 + $35 + $25
Line discount: 20% off $40 line −$8.00 Receipt-line discount fires first
Line discount: 20% off $35 line −$7.00 Receipt-line discount fires first
Total line discounts $15.00 Sum of all line-level savings
Remaining receipt total (recalculated) $85.00 $100.00 − $15.00
Whole-receipt discount: 10% off −$8.50 Applied to $85.00, NOT $100.00
Final total payable $76.50 $85.00 − $8.50

6.3 Where This Happens in the Code

The engine calls recalculateTransactionDiscounts() at two points:

Call When Why
1st After each scheme's rules are evaluated (inside the scheme loop) Ensures the transaction base is correct per-scheme before scheme totals are tallied for combination resolution.
2nd After combination resolution on the final merged result set Re-applies the transaction recalculation on the final combined discounts so the transaction base reflects all line discounts across all schemes. Only GROSS-scheme results are touched here — NET results are already cascade-calibrated and pass through unchanged.

6.4 Scheme Calculation Mode (Gross vs Net)

Each scheme carries a calculationMode field that decides which receipt total it reads when its rules are calculated. Priority, combination mode, conditions, thresholds, and line qualifiers are unaffected — only the input number changes.

The Two Modes

GROSS (default): the scheme calculates against the original receipt total — the actual sticker price before any other scheme touches it.

NET: the scheme calculates against the running total — whatever is left after all higher-priority schemes have already committed their discounts.

Default is GROSS. Existing schemes behave exactly as before unless explicitly switched to NET.

The engine walks schemes in priority-descending order against a mutable WorkingBasket. After each priority group commits its winners, line totals in the WorkingBasket are reduced. The next group's NET schemes see the reduced totals; GROSS schemes ignore the WorkingBasket and read the original receipt.

Aspect GROSS NET
Reads Original receipt total / line totals Running post-discount total / line totals
Conditions & thresholds Evaluated against original receipt Evaluated against original receipt (still gross)
Post-combination rebase Applied (today's behaviour) Skipped — values stay as calculated during the cascade
Typical use "All discounts stack on sticker price" "Each promo applies on whatever is left"

Worked example. Receipt $100, two ADDITIVE schemes:

Configuration Scheme A (prio 10, 20% off) Scheme B (prio 5, 10% off) Total off
Both GROSS 20% of $100 = $20 10% of $100 = $10 $30
A GROSS, B NET 20% of $100 = $20 10% of $80 = $8 $28
A NET, B NET 20% of $100 = $20 (nothing higher) 10% of $80 = $8 $28
A NET, B GROSS 20% of $100 = $20 10% of $100 = $10 $30
Things to Remember

• The highest-priority scheme being NET buys you nothing — nothing has committed above it yet.

• Thresholds like minOrderAmount and conditions like BASKET_VALUE always read the original receipt, so a cascade cannot disqualify itself mid-flight.

• Transaction-level discounts that get committed are prorated across lines by current-net share before the next NET scheme runs.

• BEST_OF NET candidates see the running state at evaluation time, but the winner is committed at the end of the cascade.

7. Rounding Rules, Rule Thresholds & Line Qualifiers

These three features act as additional control layers on discount rules. They are separate from conditions and entitlements and must be understood before running tests.

7.1 Rounding Rules

Every discount rule can specify how the calculated discount amount is rounded. This controls the final cent value of the discount.

Rounding Rule Behaviour Example (discount = $3.456)
NEAREST_CENT Round to 2 decimal places, half-up (standard rounding) $3.46
FLOOR Always round DOWN — favours the business $3.45
CEILING Always round UP — favours the customer $3.46
null / not set Defaults to NEAREST_CENT $3.46

7.2 Rule-Level Threshold Gates

Each discount rule has two optional threshold gates that act as a pre-check before any discount is calculated. These are different from Conditions — they are hard minimum requirements on the rule itself, not configurable condition types.

Gate Field What It Does Example
minOrderAmount The receipt total must be at least this value for the rule to fire. If below, the rule is skipped entirely. minOrderAmount = $30. Receipt $25 → rule skipped. Receipt $30 → rule proceeds.
minQuantity The total number of items on the receipt must be at least this number. If below, the rule is skipped entirely. minQuantity = 3. Receipt has 2 items → rule skipped. 3 items → rule proceeds.

7.3 Line Qualifier Gates

For LINE_ITEM and CHEAPEST_ITEM discount types, each individual receipt line is checked against two optional rule-level qualifiers before it is eligible to receive the discount. Lines that do not meet these minimums are silently skipped.

Qualifier Field What It Does Example
minLineTotal The line total (qty × unit price) must be at least this value for the line to qualify. Lines below are skipped. minLineTotal = $20. Line total $15 → skipped. Line total $20 → qualifies.
minLineUnitPrice The unit price of the item must be at least this value to qualify. Prevents discounting low-value items. minLineUnitPrice = $5. Unit price $4.99 → skipped. Unit price $5.00 → qualifies.
